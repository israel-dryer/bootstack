# Widget page template

## The model

Three kinds of section on every widget page:

- **Core** — always present, in a fixed order.
- **Catalog** — shared topical sections used when the widget has that concern. Names and ordering are fixed; presence is "if relevant."
- **Widget-specific** — idiosyncratic to this widget's capabilities. Still flat h2, just not from the shared catalog.

No `## Appearance`, `## Examples & patterns`, `## Behavior` umbrella headings. Their former subsections become flat h2s.

---

## Core scaffold

```markdown
---
title: WidgetName
---

# WidgetName

`WidgetName` [one-line description: what it is + what it's for].

=== "Usage"

    ```python
    import bootstack as bs

    app = bs.App()

    bs.WidgetName(app, ...).pack(padx=20, pady=20)

    app.mainloop()
    ```

    <div class="app-window">
        <img src="../../assets/widgets-widgetname-quickstart.png" alt="WidgetName usage example"/>
    </div>

    [CATALOG AND WIDGET-SPECIFIC SECTIONS GO HERE — pick from the catalog
    table below; follow the ordering rule]

    ## See also

    - [Guide](../../guides/path.md) — relevance to this widget

=== "API"

    ::: bootstack.widgets.primitives.widgetname.WidgetName
```

The intro line sits **above** the tabs as a page header — visible regardless of which tab is active. **Intro is description only: what the widget is and what it's for, in one sentence.** "When to use" content belongs inside the Usage tab under `## When to use`, not in the intro — repeating that material above the tabs creates a visible duplication near the top of the page.

**Tab labels:** `=== "Usage"` and `=== "API"`. The two tabs hold different *kinds* of content — hand-authored guidance vs. autogenerated reference — so the asymmetric labels are intentional. Don't use "Documentation" as a tab label; the entire site is documentation, and the redundancy adds noise without orienting the reader.

**Required tab content:**

- `=== "Usage"` — opens directly with the quickstart code sample and screenshot (no `## Usage` heading — the tab label is the section header). The first h2 inside the tab is `## When to use`. Followed by catalog and widget-specific sections in the order defined below, ending with `## See also`.
- `=== "API"` — a single mkdocstrings `:::` directive for the widget's primary class. No prose; this tab is autogenerated reference.

**Tab indentation rule:** everything inside a tab must be indented 4 spaces. This includes section headings, code fences (both the ` ``` ` lines and the code inside them), tables, HTML blocks (`<div class="app-window">`), and admonitions. Get this wrong and the content renders outside the tab.

**Cross-page tab persistence** is enabled via `content.tabs.link` in `zensical.toml`. A reader who picks "API" on one widget lands on "API" on the next. This is the power-user reading mode — comparative lookup across widgets without re-clicking the tab strip.

## Section catalog

Group the catalog entries by topic for readability below, but on the actual page every entry is a flat `## h2`.

**The "Include when" column is necessary but not sufficient.** A catalog section earns its own h2 only if it adds something beyond a one-row entry in the `## Common options` table — extra samples, a callout, edge cases, a behavior table, a screenshot. If the section would just restate the table row, drop the section and let the table carry the kwarg. This applies most often to `## Density`, `## Surface`, `## Accent and variant`, and `## State` on widgets where the kwargs are uniform and the defaults work.

### Styling

| Section heading | Include when | Scope |
|---|---|---|
| `## Accent and variant` | Widget accepts `accent=` or `variant=` | Color tokens + visual variants (solid/outline/ghost/link/text/inverse). Code samples + screenshot. |
| `## Density` | Widget accepts `density=` | `"default"` vs `"compact"`. Brief code samples. |
| `## Surface` | Widget accepts `surface=` and surface choice matters (e.g., card/overlay contexts) | Code sample showing widget inside a non-default surface. |
| `## Icons` | Widget accepts `icon=` / `icon_only=` / `on_icon=` / `off_icon=` | Icon usage, stateful icons, icon-only patterns. Link to icons guide. |
| `## Size and layout` | Widget exposes `width`, `padding`, `underline`, `anchor`, or other layout-affecting kwargs | Quick samples. Skip if defaults are universally fine. |

### Configuration

| Section heading | Include when | Scope |
|---|---|---|
| `## Common options` | Widget has 4+ constructor options not covered by other catalog sections | Table of `Option | Purpose` (or `Option | Type | Purpose`). Default to table; use h4 + code per option only for options that genuinely need an example (≤3 per page). |
| `## Value model` | Widget separates raw text from a committed value (Field-based inputs, SelectBox, etc.) | `Text` vs `Value` distinction; `.get()` / `.value` / `.set()`. Use the standard two-row table. |
| `## State` | Widget accepts `state="disabled"` / `state="readonly"` and has `disable()` / `enable()` / `readonly()` methods | Code sample showing each. |
| `## Range and stepping` | Widget accepts `minvalue` / `maxvalue` / `increment` / `wrap` | Bounded numeric widgets. |
| `## Items and values` | Widget accepts a list of selectable options (`items=` / `values=` / `options=`) | Population, addition, removal, lookup. |

### Composition

| Section heading | Include when | Scope |
|---|---|---|
| `## Add-ons` | Widget supports `insert_addon()` (Field widgets) | Prefix/suffix add-ons, named retrieval, state inheritance. Link to TextEntry add-ons reference for the full API. |
| `## Item management` | Widget manages a collection (RadioGroup, ToggleGroup, ButtonGroup, ContextMenu, etc.) | `add()`, `remove()`, `item()`, `keys()`, `configure_item()`. |
| `## Orientation` | Widget accepts `orient=` and the choice meaningfully changes layout | `"horizontal"` vs `"vertical"`. |

### Behavior

| Section heading | Include when | Scope |
|---|---|---|
| `## Events` | Widget emits any virtual events or has `on_*()` registration methods | Code sample for `bind()` / `on_*()`; events table with `Event | When it fires`. |
| `## Selection` | Widget has selectable items | `selection()`, `selection_set()`, `selection_add()`, `selection_remove()`, `selectmode` modes. |
| `## Item operations` | Widget supports operations on individual items (TreeView, ListView) | `item()`, `move()`, `delete()`, `detach()`, parent/children navigation. |
| `## Validation` | Widget supports `add_validation_rule()` (Field widgets) | Rules, triggers, validation events. Link to validation guide. |
| `## Keyboard navigation` | Widget has keyboard interaction beyond Tab + Space/Enter activation | Table of `Key | Action`. |
| `## Activation` | Widget has trigger options (ContextMenu, popovers) | `trigger=` values, `show()` / `hide()` patterns. |

### Cross-cutting

| Section heading | Include when | Scope |
|---|---|---|
| `## Reactivity` | Widget accepts `signal=` or `textsignal=` | Signal binding example; subscribe/unsubscribe; link to reactivity guide. |
| `## Localization` | Widget renders user-visible strings and accepts `localize=` | Brief gettext-key example; link to localization guide. |

## Ordering rule

Sections appear in this order. Skip any that don't apply; widget-specific sections slot in where they topically belong.

1. **Core opening**: intro line above the tabs, then inside the Usage tab — quickstart code sample + screenshot (no `## Usage` heading), then `## When to use` (with the `### Consider a different control when...` h3).
2. **Styling**: `## Accent and variant` → `## Density` → `## Surface` → `## Icons` → `## Size and layout`. Widget-specific styling concerns (e.g., `## Per-row color with tag_configure`) slot here.
3. **Configuration**: `## Common options` → `## Value model` → `## Items and values` → `## Range and stepping` → `## State`. Widget-specific config concerns slot here.
4. **Composition**: `## Add-ons` → `## Item management` → `## Orientation`. Widget-specific composition concerns slot here.
5. **Behavior**: `## Events` → `## Selection` → `## Item operations` → `## Validation` → `## Keyboard navigation` → `## Activation`. Widget-specific behavioral concerns slot here.
6. **Cross-cutting**: `## Reactivity` → `## Localization`.
7. **Core closing**: `## Additional resources`.

## Widget-specific sections

When the widget exposes a concern not in the catalog, write a flat `## h2` for it and slot it into the topically closest band above. Examples:

- TreeView: `## Per-row color with tag_configure` (Styling band)
- DateEntry: `## Range selection` (Configuration band), `## Picker behavior` (Behavior band)
- ListView: `## Datasource` (Configuration band)
- ScrolledText: `## Scrollbar visibility` (Styling band)

**Rule**: if a widget-specific section turns out to apply to 3+ widgets, promote it to the catalog instead.

**Don't reinvent catalog names.** If a topic matches a catalog entry, use the catalog name. Don't write `## Triggering changes` when `## Events` covers it.

## Style conventions

### Heading depth

- `# WidgetName` (h1) — once, top of page
- `## Section` (h2) — every section, flat
- `### Subsection` (h3) — only allowed for `### Consider a different control when...` under `## When to use`, the `### Related widgets` / `### Framework concepts` / `### API reference` under `## Additional resources`, and grouped option tables under `## Common options` (e.g., when 15+ options benefit from labeled groups)
- `#### Detail` (h4) — only inside `## Common options` when an option needs an inline code example; avoid otherwise

### Common options: table or h4 + code?

**Default to table.** Use h4 + code only when the option's behavior cannot be conveyed in a one-line description (typically: `value_format`, `command` callbacks, `validation` rules, `signal` binding).

Rule of thumb: if 4+ options on a page would need h4 + code, the page is doing too much — split the complex options into their own catalog sections (`## Validation`, `## Value model`, `## Reactivity`) and leave `## Common options` as a clean reference table.

### "Consider a different control when..." format

```markdown
### Consider a different control when...

- condition-first lowercase clause — use [Widget](path.md)
- next condition — use [OtherWidget](path.md)
```

h3 heading. Three-dot ellipsis (not `…`). Bullets start lowercase, condition first, em-dash separator, action with linked widget name. No bold prefixes, no trailing "instead."

### Code samples

- Always use `bs.*` imports
- The quickstart sample (first code block in the Usage tab) is self-contained: includes `import bootstack as bs`, `app = bs.App()`, and `app.mainloop()`
- Subsequent samples may assume `app` exists; do not repeat the boilerplate
- Use `# comments` sparingly, only when behavior is non-obvious

### Quickstart shape

- ≤ 30 lines including imports and `mainloop()`
- Show the widget's primary use case, not exhaustive features
- One screenshot per quickstart, always

### Screenshots

- Path: `docs/assets/widgets-<widget>-<purpose>.png`
- Wrap in `<div class="app-window"> <img ... /> </div>`
- One per Usage (required); additional screenshots only when essential

### Callouts

- `!!! note "Title"` — clarifications, version-specific behavior, gotchas
- `!!! tip "Title"` — non-obvious patterns
- `!!! warning "Title"` — known issues, deprecated parameters
- `!!! link "..."` — cross-references to guides

Callout body ≤ 4 lines. Longer content belongs in its own section.

## Docstring conventions

The API tab renders the widget's Python docstrings via mkdocstrings. Docstrings serve a different audience than the Usage tab: IDE hover, signature help, and quick "what is this?" discovery. Optimize them for that — not for cookbook reading.

**Guiding principle.** Docstrings answer *what exists*. The Usage tab answers *how to use it*. If the same fact lives in both places, one will drift.

### Keep in docstrings

- **Class summary** — one sentence on what the widget is.
- **Class description** — one short paragraph; enough to orient an IDE reader.
- **`Args:` on `__init__`** — the constructor contract. Each parameter, one line. This isn't duplicated by the Usage tab's `## Common options` table — the table is curated/grouped, while `Args:` is the formal signature reference. Both have a job.
- **`Attributes:`** — *only* for attributes that wouldn't otherwise be visible to griffe (runtime-assigned in `__init__` like `variable`, `signal`). Properties (`@property`) get their own headings via mkdocstrings — don't list them in Attributes too.

### Move to the Usage tab

- **`!!! note "Events"` admonitions** — Usage tab has `## Events`. Duplicates.
- **`Example:` / `Examples:` blocks** — Usage tab has the quickstart and pattern-specific sections.
- **Long behavioral descriptions** — Usage tab has catalog sections (`## Selection`, `## Activation`, `## Keyboard navigation`, etc.).
- **"See also" / "Related" sections in docstrings** — Usage tab has `## See also`.
- **Tutorial prose explaining when to use it** — Usage tab has `## When to use`.

### What good looks like

```python
class WidgetName(BaseClass):
    """One-sentence summary of what the widget is and what it's for.

    One short paragraph of orientation: what it renders, what it accepts,
    what it emits. No cookbook content — that lives in the Usage tab.

    Attributes:
        signal (Signal): Runtime-bound signal (not visible to griffe otherwise).
        variable (Variable): Runtime-bound Tk variable.
    """

    def __init__(self, master, *, value=None, items=None, **kwargs):
        """Create a WidgetName.

        Args:
            master: Parent widget.
            value: Initial value.
            items: Sequence of selectable options.
        """
```

## Pre-publish checklist

- [ ] All h2s are either core, catalog (verbatim names), or widget-specific
- [ ] Catalog sections appear in the order defined above; widget-specific sections slot in topically
- [ ] No `## Appearance`, `## Examples & patterns`, or `## Behavior` umbrella headings
- [ ] No catalog-name shadowing (e.g., `## Triggering changes` where `## Events` would do)
- [ ] Every required core section is present; no stubbed sections with placeholder content
- [ ] Usage ≤ 30 lines and runs from a clean Python
- [ ] All code samples use `bs.*` imports
- [ ] Screenshots exist and paths resolve
- [ ] All `[Link](path.md)` references point to real files (zensical link validation is OFF — grep manually)
- [ ] `### Consider a different control when...` follows the canonical format
- [ ] Event names match source (`<<Change>>` not `<<Changed>>`, etc.)
- [ ] `signal=` vs `textsignal=` matches what the widget accepts
- [ ] `## Common options`, if present, is a table — unless ≤ 3 options genuinely need code examples (then h4 + code is acceptable for those alone)
- [ ] `## See also` has ≥ 2 related widgets, ≥ 1 framework concept
- [ ] Tab labels are `=== "Usage"` and `=== "API"` (not "Documentation" / "API")
- [ ] No `## Usage` heading inside the Usage tab — the quickstart code + screenshot lead, and the first h2 is `## When to use`
- [ ] No catalog section (e.g., `## Density`, `## Surface`) restates a Common options table row without adding samples, callouts, edge cases, or a screenshot
- [ ] Widget-specific behavioral sections (search modes, custom-value modes, etc.) precede catalog-band Behavior sections (`## Events`, `## Validation`, `## Keyboard navigation`) when they define the widget's core variants
- [ ] Class docstring contains no `!!! note "Events"`, no `Example:`, no "See also" — those duplicate Usage tab content (see Docstring conventions)
- [ ] `Attributes:` lists only runtime-assigned names invisible to griffe (signal, variable), not @property declarations
